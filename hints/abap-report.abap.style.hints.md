# abap-report Style Hints

These hints apply to all ABAP source generated by the `abap-report` template.
They reflect modern ABAP (NetWeaver 7.57+, ABAP Cloud) idioms.

This file is the **fifth hints layer** in PCD: `<scope>.<language>.style.hints.md`.

---

## 1. Naming

### 1.1 Variables

| Prefix | Scope                          | Example          |
|--------|--------------------------------|------------------|
| `lv_`  | local variable, scalar         | `lv_count`       |
| `lt_`  | local table                    | `lt_prices`      |
| `ls_`  | local structure                | `ls_entry`       |
| `lo_`  | local object reference         | `lo_app`         |
| `lr_`  | local data reference           | `lr_target`      |
| `lf_`  | local field symbol             | `lf_row`         |
| `mv_/mt_/ms_/mo_` | instance attribute  | `mt_prices`      |
| `gv_/gt_/gs_/go_` | report-level global | `go_app`         |
| `iv_/it_/is_/io_` | importing parameter | `iv_country`     |
| `ev_/et_/es_/eo_` | exporting parameter | `ev_result`      |
| `cv_/ct_/cs_/co_` | changing parameter  | `ct_buffer`      |
| `rv_/rt_/rs_/ro_` | returning parameter | `rv_count`       |
| `c_`              | constant            | `c_glass_size_l` |

Instance globals (`go_`) at report level are tolerated but discouraged.
Prefer constructor injection.

### 1.2 Types

- Local record type: `ty_<name>` — singular noun (`ty_price_entry`)
- Local table type: `tt_<name>` — plural noun (`tt_price_table`)
- Global interface: `zif_<name>`
- Global class: `zcl_<name>`
- Global exception: `zcx_<name>`

### 1.3 Methods

- Lowercase, snake_case, verb-first
- Bad: `country_validation`. Good: `validate_country`.
- Bad: `do_estimate`. Good: `estimate_bees`.
- Method body ≤30 lines. If longer, extract.
- Public methods on the application class correspond 1:1 to spec BEHAVIORs;
  do not invent additional public methods.

### 1.4 Length-limit abbreviations

When a name would exceed an ABAP length limit, abbreviate in this order of
preference:

1. Drop vowels from the middle of words: `BeeCount` stays, `MeasurementValue` →
   `MsrmntValue` only if needed.
2. Use established SAP abbreviations: `MGMT`, `CFG`, `MSG`, `ATTR`, `VAL`.
3. Acronyms are uppercase in names but lowercase in identifiers.

Never abbreviate the first word — `ZCL_C_SK_BDJVC` is unreadable. Prefer
shortening the spec name itself if necessary.

## 2. Modern ABAP idioms

### 2.1 Inline declarations

Use `DATA(...)` and `FINAL(...)`:

```abap
DATA(ls_price) = price_for_country( 'CZ' ).
FINAL(lv_rate) = rate_for_currency( ls_price-currency ).
```

`FINAL(...)` (NetWeaver 7.57+) communicates write-once intent. Use it
whenever the variable is not reassigned after declaration.

Avoid top-of-method `DATA:` blocks except for accumulator variables that
need an initial value.

### 2.2 Value constructors

```abap
mt_prices = VALUE #(
  ( country = 'AT' currency = 'EUR' price = '9.00'   )
  ( country = 'CZ' currency = 'CZK' price = '100.00' )
).
```

Never `APPEND … TO …` in a loop to build a static table.

### 2.3 Conditional expressions

```abap
DATA(lv_label) = COND string(
  WHEN iv_count = 1 THEN `bee`
  ELSE                   `bees` ).
```

Use `COND` for inline conditional values. Use `SWITCH` when dispatching on
an enum-like value. Keep an explicit `ELSE`.

### 2.4 String templates

```abap
lv_msg = |Count for { lv_country } is { lv_bees DECIMALS = 1 }.|.
```

Use over `CONCATENATE`. Numeric formatting controls (`DECIMALS`, `WIDTH`,
`ALIGN`) go inside the `{ }`.

### 2.5 Table expressions

```abap
DATA(ls_row) = lt_table[ key = value ].          " raises if missing
DATA(ls_row) = VALUE #( lt_table[ key = value ] OPTIONAL ). " empty if missing
```

Use over `READ TABLE … WITH KEY …` for read-only access. The `READ TABLE`
form remains correct when you need `sy-tabix` or `sy-subrc` afterwards.

### 2.6 LOOP/FOR

For pure transformation, use `FOR`:

```abap
DATA(lt_codes) = VALUE string_table(
  FOR ls IN mt_prices
  ( CONV string( ls-country ) ) ).
```

For side effects (e.g. WRITE), use `LOOP AT`.

## 3. Forbidden patterns

| Pattern                                    | Why                              |
|--------------------------------------------|----------------------------------|
| `FORM ... ENDFORM`                         | Pre-OO. Use methods.             |
| `TABLES <ddic_table>` at report level      | Implicit globals. Use local data.|
| `DATA: BEGIN OF ... END OF` (in code)      | Use TYPES BEGIN OF, then DATA.   |
| `WITH HEADER LINE`                         | Implicit work area, deprecated.  |
| `MOVE-CORRESPONDING` on unrelated structs  | Use `CORRESPONDING #( ... )`.    |
| `IF NOT lv_x IS INITIAL`                   | Use `IF lv_x IS NOT INITIAL`.    |
| `CHECK` mid-method                         | Use early-return at top.         |
| `EXIT` from a loop without a clear reason  | Use `WHILE` or table expression. |
| Header line on internal tables             | Always full work areas.          |

## 4. String literals

- Single quotes `'...'` for `TYPE c` values. Trailing blanks are trimmed.
- Backticks `` `...` `` for `TYPE string` values. Trailing blanks preserved.
- Always pick the right delimiter for the target type — mixing causes silent
  truncation on assignment from `c` to `string`.

```abap
DATA lv_text TYPE string VALUE `Hello`.     " correct
DATA lv_text TYPE string VALUE 'Hello'.     " also works but conceals intent
DATA lv_code TYPE c LENGTH 2 VALUE 'CZ'.    " correct
```

## 5. Comments

- Comment placement: line above the statement.
- `*` at column 1 only for major section banners — section headers above
  TYPES / methods.
- `"` at any column for inline notes.
- No TODO comments without a tracking ID and an owner.
- Do not paraphrase the code in comments. Comment **why**, not **what**.

## 6. Field-symbols vs references

| Use                                       | Choice            |
|-------------------------------------------|-------------------|
| Iterate a table to mutate rows in place   | `FIELD-SYMBOLS`   |
| Pass data into / out of methods           | `REF TO <type>`   |
| Cache a complex sub-structure within a method | `FIELD-SYMBOL` is fine |

Never combine `FIELD-SYMBOL TYPE ANY` with dynamic `ASSIGN COMPONENT`
unless the spec genuinely requires reflection. If it does, raise a flag
to the spec author — reflection is rarely what was meant.

## 7. Error handling

| Error class                       | Base class            | Use for                    |
|-----------------------------------|------------------------|----------------------------|
| Domain (anticipated, caller-fixable) | `cx_static_check`   | Spec ERRORs                |
| System (DB lost, RFC dropped)     | `cx_dynamic_check`     | Infrastructure failures    |
| Programming (assertion violated)  | `cx_no_check`          | Should-never-happen        |

The translator emits a single `ZCX_<NAME>` exception class for the package,
inheriting from `cx_static_check`. All spec ERRORs are raised through it
with `token` carrying the spec-level identifier.

Never check `sy-subrc` more than two levels deep. If you find yourself
checking `sy-subrc` after every statement, extract a wrapper method that
returns a typed result.

## 8. Clean core

- Only released APIs (verifiable in SAP API Hub).
- Customer namespace (`Z*`/`Y*` or registered).
- No enhancement implementations against non-released enhancement spots.
- No object modifications.
- No `INSERT REPORT`, no dynamic table generation.
- ABAP Cloud development model preferred; classic ABAP only when the spec
  explicitly opts out (`Clean-Core: false` in META).

## 9. Spec hash (PCD rule A.20)

The application class MUST declare:

```abap
CONSTANTS c_spec_hash TYPE string VALUE
  `<64-character hex SHA256 of the spec source>`.
```

This constant is consumed by:
- The synthesized `version` operation (writes hash to list).
- The report documentation (first 12 chars of the hash).
- The package README.md.

The translator computes the hash from the spec source file at translation
time. RULE-18 in pcd-lint detects drift between the embedded hash and the
current spec.

## 10. ABAP Unit conventions

- Test classes go in `*.clas.testclasses.abap` (abapGit convention).
- Test class name: `LTC_<BEHAVIOR>` for behaviour tests, `LTC_INVARIANTS`
  for invariant tests, `LTC_<INTERFACE>_<DOUBLE>` for doubles.
- `DURATION SHORT` and `RISK LEVEL HARMLESS` unless the BEHAVIOR genuinely
  needs longer-running or higher-risk fixtures (rare).
- `setup` method creates a fresh `mo_cut` for every test method.
- `cl_abap_unit_assert=>assert_equals` for value comparisons.
- `cl_abap_unit_assert=>assert_that` with a matcher for richer predicates.
- Never `cl_abap_unit_assert=>fail` without a message.

## 11. ABAP Cloud-specific restrictions

When `Clean-Core: true` (default), the translator MUST additionally avoid:

- `OPEN DATASET` (no application server filesystem access)
- `WRITE TO DATASET`
- `KERNEL_CALL`
- Function modules outside the released API set
- Anything that requires `S_DEVELOP` at runtime

If the spec requires any of these, set `Clean-Core: false` in META — this
blocks deployment to BTP ABAP Environment but allows S/4HANA private cloud
and on-premise.
