# Hints Files

A **hints file** carries implementation knowledge that belongs neither in
the specification nor in the deployment template. Specifications are
language-neutral by design; templates cover language and packaging
conventions, not library internals. Library API shapes, scaffold-pass
patterns, and project-specific coding conventions live here.

Hints are advisory. They cannot override a specification `INVARIANT` or a
template constraint. A translator that follows a hints file produces
better output; a translator that ignores one still produces correct
output if the spec and template are sufficient.

---

## The Five Layers

PCD recognises five kinds of hints file, each with a distinct scope and a
distinct location in the preset hierarchy. The kind is encoded in the
filename — the toolchain parses it, so the convention is load-bearing.

### 1. Milestone hints

```
<template>.<language>.milestones.hints.md
```

Scaffold-first patterns for milestone-based translation. These files
describe the structural decisions that should be locked in during the
scaffold milestone (M0): package layout, file naming, interface shapes,
stub conventions, build target structure. They are reusable across every
component that uses the same template and language.

Read by the translator at the start of the scaffold milestone, before any
source file is written.

Examples in this directory: `cli-tool.go.milestones.hints.md`,
`cli-tool.rs.milestones.hints.md`,
`kubectl-style-cli.go.milestones.hints.md`.

### 2. Library hints

```
<template>.<language>.<library>.hints.md
```

Verified API shapes, exact version pins, and known gotchas for one
specific library in one language and template context. Library hints
exist because LLM translators routinely fabricate plausible-looking API
calls that compile but fail at runtime, or pick deprecated alternatives
when a current one exists.

The `mcp-server.go.mcp-go.hints.md` file is the canonical example: it
records that the correct constructor for streamable-HTTP servers is
`NewStreamableHTTPServer`, not the deprecated `NewSSEServer`, and that
domain errors must be returned via `NewToolResultError`, not Go `error`
values. Without this file, translators reliably produce code that fails
in subtle ways.

Examples in this directory: `mcp-server.go.mcp-go.hints.md`,
`cloud-native.go.go-libvirt.hints.md`,
`cloud-native.go.golang-crypto-ssh.hints.md`.

### 3. Implementation hints

```
<component>.implementation.hints.md
```

Component-specific, language-neutral implementation knowledge. File
groupings, required field names, known failure modes, domain conventions
that any implementation in any language must respect.

Implementation hints sit next to the spec they relate to, not in this
directory, because their scope is one component.

### 4. Style hints

```
<scope>.<language>.style.hints.md
```

Coding style and architectural conventions for one language within one
scope. The scope is a project name or a company name, and the file lives
at the matching level of the preset hierarchy:

- `/etc/pcd/hints/suse.go.style.hints.md` — company-wide Go style; applies
  to every Go translation on the machine.
- `.pcd/hints/myproject.go.style.hints.md` — project-specific style;
  applies only within that project.

Style hints address the standard objection that AI-generated code cannot
adhere to established project conventions. With a style hints file, it
can: naming standards, framework idioms, forbidden patterns, and
architectural preferences are read by every translator that runs against
that project or under that organisation.

Style hints are authored once and maintained over time. They are
*declarative*, not generated.

### 5. Decisions hints

```
<specname>.<language>.decisions.hints.md
```

Implementation decisions made by a previous translation run that are not
inferable from the specification alone. Where the scaffold split a file
because it grew too large; which of two valid library choices was picked;
how the renderer was factored. These decisions are correct but
under-determined — a fresh translator would make different, equally valid
choices.

Decisions hints files are *generated by the translator*, not authored by
hand. They live next to the specification, not in this directory, because
they are spec-scoped and disposable: switching the target language from
Go to Rust invalidates the entire file.

Decisions hints are the mechanism that makes guided regeneration possible
without contaminating the specification with implementation details that
would lock it to one language.

## Naming Convention as Contract

The filename pattern is not documentation — it is parsed by
`mcp-server-pcd` and consumed by translators. A hints file with the
correct content but the wrong filename will not be loaded.

Three rules:

1. The first segment names the template (for milestone, library, and
   style hints) or the component (for implementation hints) or the
   specification (for decisions hints).
2. The second segment, when present, names the language using its
   conventional short form: `go`, `rs`, `py`, `c`, `cpp`, `cs`.
3. The middle segment names the library or the file kind:
   `milestones`, `implementation`, `style`, `decisions`, or the library
   name.

A hints file that targets one language only — where the template name
already implies the language — may omit the language segment. The
`python-tool.hints.md` file in this directory does exactly this, because
`python-tool` is a Python-only template and the language is redundant.

## The Preset Hierarchy

Hints files follow the same layering model as deployment templates and
presets:

1. Vendor — `/usr/share/pcd/hints/` (this directory, as installed by the
   `pcd-templates` package)
2. System — `/etc/pcd/hints/` (organisation-wide additions and overrides)
3. User — `~/.config/pcd/hints/` (individual additions and overrides)
4. Project — `.pcd/hints/` (project-local, version-controlled)

A hints file present at a later layer overrides any file of the same
name at an earlier layer. This means an organisation can ship its own
verified API shapes for an internal fork of a library; a project can
pin a different library version for one specific translation. The
upstream file is not touched.

## Adding a New Hints File

A new hints file is appropriate when:

- A translation run has failed or produced wrong output, and the cause
  was missing knowledge about a library, a scaffold pattern, or a project
  convention.
- The same knowledge would be needed for any future translation of the
  same template and language combination.
- The knowledge does not belong in the specification (because it is
  language-specific or implementation-specific) and does not belong in
  the template (because it is library- or project-specific).

Hints files should be concise. The point is to give the translator the
single fact it needs, not to teach the underlying topic. Code snippets
showing the right and wrong pattern are usually clearer than prose. See
the existing files for tone and length.

A hints file is not a substitute for fixing a defect in the template or
the prompt. If translators systematically get something wrong across many
libraries, the fix belongs upstream — in the template's `EXECUTION`
section or in `prompts/prompt.md` — not in a per-library workaround.
